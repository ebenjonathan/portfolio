Stop Doing This Work Manually in 2026

Jonathan Mupini November 2025 6 min read

At Traverze Travel, every booking started with an email or a phone call. Staff opened three separate tools: a GDS terminal for flight and hotel availability, a spreadsheet to record booking details, and an email client to send confirmations. Each booking took between 20 and 40 minutes to process.

The problem was not the people. The problem was the tools. Nothing connected to anything else. Data was typed twice. Errors crept in when details moved between systems.

The Problem in Detail

A typical booking looked like this:

  1. Receive request by email or phone
  2. Open the GDS terminal and check flight availability manually
  3. Copy passenger details from the email into the spreadsheet by hand
  4. Write and send a confirmation email from scratch
  5. Update the spreadsheet again after payment came in

Every step was manual. There was no validation. Customers sometimes received confirmations with wrong dates or wrong pricing. The spreadsheet had no audit trail. If something went wrong three weeks after booking, there was no clean record to refer to.

What I Built

I built an internal web application with three main components:

1. A single booking intake form

Staff enter all booking details in one place. The form validates input before submission. No more partial records. No more copying between tabs. Passenger names, travel dates, contact details, and payment status all live in one record from the start.

2. A live availability layer

The form connects to a third-party travel API. Staff type a destination and see current flight and accommodation options without switching to the GDS terminal. Prices and availability come from the API in real time.

3. Automated confirmation emails

When a booking is confirmed, the system generates a formatted confirmation email and queues it for review. Staff approve and send in one click. No copying. No formatting. No blank fields.

Technical Approach

The frontend was built with JavaScript. The backend ran on Node.js with a REST API structure. Data lived in a PostgreSQL database hosted on Supabase. The app deployed to Vercel.

The GDS API integration was the hardest part. Most travel APIs are designed for enterprise systems. They return large XML or JSON payloads with fields the frontend did not need. I wrote a normalisation layer that stripped the response down to what the booking form needed: routes, prices, seat availability, and booking reference.

I also hit API rate limits early. The availability search was triggering on every keystroke in the destination field. I added a 400ms debounce on the input and a server-side cache for recent results. This reduced API calls by roughly 60% without changing the user experience.

// Debounced search, avoids hitting the API on every keystroke
const searchAvailability = debounce(async (query) => {
  const cached = await cache.get(query);
  if (cached) return cached;
  const result = await travelApi.search(query);
  await cache.set(query, result, { ttl: 300 });
  return result;
}, 400);

Retry logic was added to every outbound API call. Travel APIs go down. A booking system that fails silently is worse than one that shows an error. Every failed API call was logged with the full request context.

Results

Outcomes

  • Booking processing time: 20...40 minutes ? under 5 minutes
  • Manual data entry errors: near zero (form validates all required fields)
  • Team hours recovered per week: 10+
  • GDS and third-party manual touchpoints reduced by ~35%

What I Learned

Start with the data flow. I spent two weeks designing screens before I mapped every field and where it came from. When I finally drew the data flow, three of my planned screens collapsed into one.

A unified interface beats three separate tools. Each individual tool the team used was capable. But switching between them cost time and caused copy errors. One simple interface with fewer features won immediately.

Build retry logic on day one. The travel API returned partial responses under load. I found this in production on day three. Adding retry logic after the fact takes twice as long as building it in from the start.
