No Data. No Money. Stuck Abroad.

Jonathan Mupini September 2025 8 min read

In 2022, on my first flight out of Zimbabwe to South Africa, I experienced one of those situations that quietly exposes how fragile our digital dependency is.

My Mastercard was not working. Mobile roaming failed. I could not contact my uncle who was supposed to pick me up. And all my funds were behind an OTP system that required email access I could not reach.

I was effectively stranded.

So I did what developers naturally do. I started debugging the system.

The System Failure

At a high level, this is what broke down:

[ Bank Account ]
      ↓ (OTP via email)
[ Email Access ] X No internet
      ↓
[ Mobile Roaming ] X Failed
      ↓
[ Airport Pickup ] X No communication
      ↓
[ Physical World Problem ]

This was not just a travel issue. It was a cascading dependency failure in a connected system. One missing layer broke everything.

First Attempts (and Failures)

I tried the obvious solutions:

1. Mobile Roaming

  • Enabled roaming
  • No connection

2. Cafe WiFi

  • Weak signal outside
  • Encrypted network with client-only access
[Cafe WiFi]
     |-- Strong inside only X
     |-- Auth required X
     `-- Not accessible X

At this point, I had connectivity options, but none were usable.

The Hidden Layer: Captive Portals

Most public WiFi networks are not open internet. They are controlled networks that require authentication before granting access. This system is called a captive portal.

Typical flow

Device connects to WiFi
        ↓
Gets IP address
        ↓
But internet is blocked
        ↓
Redirected to login page (captive portal)
        ↓
User authenticates
        ↓
Internet granted

Sometimes the redirect never triggers automatically. That is where most users get stuck.

The Breakthrough: Forcing the Portal

I remembered two tools that force captive portal detection:

  • http://neverssl.com
  • http://detectportal.firefox.com

Why This Works (Technical Explanation)

1. HTTP vs HTTPS behavior

Modern browsers default to HTTPS, which is secure and does not allow interception easily. Captive portals rely on intercepting unsecured HTTP traffic.

HTTPS request
-------------
Browser -> Encrypted -> Server
X Cannot be intercepted easily

HTTP request
------------
Browser -> Plain text -> Network -> Redirect possible
OK Captive portal can hijack this

2. neverssl.com

This endpoint is always HTTP, has no HTTPS fallback, and avoids automatic redirects. That makes it ideal for triggering captive portals.

User opens site
      ↓
Network intercepts HTTP request
      ↓
Redirects to WiFi login portal
      ↓
Login page appears

3. detectportal.firefox.com

This URL is used by Firefox and many operating systems for connectivity checks. If it gets redirected, the device detects a captive portal and launches login UI.

Device -> Test URL request
        ↓
If redirected -> Captive portal detected
        ↓
System triggers login UI

What Happened Next

Once I used one of these URLs, the captive portal finally appeared. I was able to log into airport WiFi, access my mobile carrier portal, reactivate international roaming, restore email access, resolve OTP authentication, and contact my uncle.

System restored.

Architecture Lesson: Dependency Chains in Real Life

This incident looked like a travel problem, but it was actually a distributed system failure across identity, connectivity, and authentication layers.

Identity (Bank)
    ↓
Authentication (OTP via Email)
    ↓
Network Access (Mobile / WiFi)
    ↓
Captive Portal System
    ↓
Physical coordination (pickup, travel)

One broken node collapsed the entire system.

Practical Guide: Connecting to Public WiFi (Developer Checklist)

  1. Connect to WiFi even if it shows No Internet.
  2. Force HTTP traffic with http://neverssl.com.
  3. Trigger OS captive detection with http://detectportal.firefox.com.
  4. Complete portal login: accept terms, enter voucher or OTP if required, and authenticate session.
  5. Avoid HTTPS sites during login phase because they often block portal redirection.

Mental Model for Developers

[ WiFi Network ]
      ↓
[ Firewall Layer ]
      ↓
[ Captive Portal Gatekeeper ]
      ↓
[ Internet Access ]

You are never online immediately. You are connected to a network, but not yet authorized.

Key Takeaway

Public WiFi is usually not broken when it does not work immediately. It is often waiting for you to trigger the correct authentication path. Sometimes the simplest debugging tool is knowing how to generate the right HTTP request.
